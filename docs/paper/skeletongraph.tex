\begin{abstract}
A wave of code-retrieval tools advertises dramatic token savings---99\% (Codebase-Memory),
70$\times$ (Graphify), 38--528$\times$ (code-review-graph)---but those numbers are
\emph{token-count comparisons against reading whole files}, not measurements of whether the
agent actually fixes more bugs, or fixes them more cheaply, once the savings pass through a
real agentic loop. We close that gap. We build a controlled \emph{systems} harness in which
every retrieval pipeline keeps its \emph{native} tools (graph queries, repo-map injection,
function fetch) while sharing one substrate---the same model, tasks, edit/submit interface,
and SWE-bench verifier---and we run eight pipelines on SWE-bench Verified.
We find that (1) on these popular, pre-cutoff repositories \emph{solve rate is nearly
insensitive to retrieval}: a no-retrieval agent scores $37.0\%$ and no pipeline is
significantly better; (2) the headline token claims \emph{do not survive the loop}---an agent
given an imprecise map simply reads more files, so end-to-end cost converges, and a tool that
front-loads a large repo map (aider) costs $7\times$ more, not less; and (3) what actually
differentiates pipelines is \emph{retrieval correctness at function granularity} together with
\emph{token-cheap fetch}. Our method, \sg{}---a tree-sitter structural index used to
\emph{rerank} cheap lexical retrieval and to fetch one function at a time---delivers the best
file and function recall of any arm while using $\sim$35\% fewer tokens than every baseline and
competitor, at a comparable solve rate. Notably, \sg{} builds its index with \emph{no LLM}---as does the tree-sitter competitor
Codebase-Memory---while Graphify pays a per-repository LLM graph build that the token-only
benchmarks omit; against the zero-LLM competitor, \sg{} still retrieves more precisely at lower
token cost, isolating \emph{retrieval quality}, not build cost, as the difference.
Our framing inverts the field's. The recent wave of graph-based tools reduced code context to a
\emph{token-optimization} problem; we argue the goal is \emph{better retrieval}---landing the
correct function---of which lower token cost is a downstream consequence that is only visible,
and only real, once measured end-to-end inside the agent loop.
We release the harness and three methodology fixes uncovered along the way.
We additionally deploy \sg{} as a Model Context Protocol (MCP) tool for a production agent
(Claude Code) and introduce a \emph{retrieval-first} hook that enforces index use---a lever
no competing tool provides---finding that for a strong, natively-equipped agent the index's
value surfaces as efficiency (fewer turns, smaller peak context) rather than solve rate,
consistent with our harness results.
\end{abstract}

\section{Introduction}
Modern coding agents (SWE-agent, OpenHands, Aider, Claude Code) spend most of their budget
\emph{reading code}. A small industry of retrieval tools promises to cut that budget by
indexing the repository into a graph or map and serving compact context. The advertised
reductions are eye-catching---Codebase-Memory cites ``99\% fewer tokens,'' Graphify ``70$\times$,''
code-review-graph ``38--528$\times$.'' Each of these numbers, however, is computed the same way:
\emph{tokens to answer one query from the index} versus \emph{tokens to read every source file}.
None measures the quantity a practitioner cares about---\emph{does the agent resolve more issues,
and at what real end-to-end cost?}

This matters because an agent is a feedback loop, not a single query. If retrieval is imprecise,
the agent simply issues more searches and reads more files until it finds the fix; the
per-query saving is recovered by the loop, and total cost converges. Conversely, a tool that
front-loads a large static map pays for it on \emph{every} turn. The only way to know whether a
retrieval method helps is to put it inside the loop and measure the whole run.

We therefore re-center the question. The interesting axis is not ``how few tokens does the index
return'' but ``how often does the agent end up looking at the \emph{right function}.'' Token cost
follows from that: a retriever that lands the correct function is read once and cheaply; a
retriever that does not sends the agent back to read more, and the advertised saving evaporates.
\sg{} is built around this principle---structure to rank and to fetch one function at a time,
cheap lexical breadth for recall---rather than around a token-count headline.

We do exactly that. Our contributions:
\begin{enumerate}
\item A \textbf{fair systems harness} in which each pipeline exposes its \emph{native} tools
(\sg{}: structural search + function fetch + graph expand; cbmem: 14 graph tools; aider:
PageRank repo-map injected into context; graphify: graph query/explain), sharing only the
model, tasks, edit/submit interface, and SWE-bench verifier. Baselines (BM25, ripgrep, dense
hybrid, no-retrieval) keep the standard tool set.
\item An \textbf{empirical finding} that on contaminated benchmarks solve rate is
retrieval-insensitive, and that the popular token-reduction claims do not survive the agentic
loop. We make this concrete with a per-turn cost decomposition and a contamination probe.
\item \textbf{\sg{}}, a tree-sitter structural index deployed as a \emph{reranker} over cheap
lexical retrieval plus a \emph{function-level fetch} tool. \sg{} achieves the best retrieval
quality (file and function recall, rank) of any arm at the lowest token cost, with \emph{zero}
LLM calls to build its index.
\item Three reusable \textbf{methodology fixes} (per-arm SWE-bench verification to avoid shared
verdicts, an edit-thrash circuit-breaker, and retrieval-cache patch-leakage prevention) and an
open harness.
\item A \textbf{production deployment} of \sgr{} as an MCP server with a \emph{retrieval-first}
\texttt{PreToolUse} hook that gates native text search behind the structural index, with a
bounded fallback so a task the index cannot localise never deadlocks. To our knowledge no
surveyed tool enforces tool preference; they expose the index and rely on the model to prefer
it. We report a preliminary in-the-loop evaluation on Claude Code (\S\ref{sec:deploy}).
\end{enumerate}

\section{Related Work and the Claims We Test}
\textbf{Token-reduction tools.} Codebase-Memory, Graphify, and code-review-graph each build a
code knowledge graph and report large token reductions, validated by token-count math rather
than task resolution. Graphify additionally requires an LLM to construct its graph (semantic edge
inference), a one-time cost that a per-task agent re-pays on every repository; Codebase-Memory, as
we evaluate it, builds a tree-sitter knowledge graph with no LLM---a clean zero-LLM structural
control for \sg{}. \textbf{Repo maps.} Aider injects a PageRank-ranked tree-sitter map into the
prompt; the model navigates from it. \textbf{Retrieval-augmented code generation.}
CodeRAG-Bench reports that retrieval quality and generation success are positively but
\emph{not straightforwardly} correlated---retrievers struggle to fetch useful context and
generators struggle to use it; we observe the same. \textbf{Benchmark contamination.} SWE-bench
Verified comprises pre-cutoff issues from popular repositories; recent work shows models can
name the buggy file from the issue text alone at $\sim$76\% accuracy, so a no-retrieval agent is
a deceptively strong baseline. We treat this as a first-class confound rather than a footnote.
A related but distinct source of inflation is \emph{prompt-level symbol leakage}: many SWE-bench
issues contain the function name, class name, or full file path verbatim---in tracebacks, code
snippets, or prose---so any lexical retrieval system (BM25, grep, \sg{}, cbmem, graphify) can
match on those literals without performing structural inference. This affects all arms
symmetrically; we discuss its impact on relative comparisons in \S\ref{sec:analysis}.

\section{SkeletonGraph}
\sg{} parses a repository with tree-sitter into a \emph{skeleton table} (every function/class
with its signature, span, and file) and a \emph{call graph}. No LLM is used. From this index it
exposes three agent tools beyond the shared \texttt{edit}/\texttt{submit}:
\begin{itemize}
\item \texttt{search\_code}: structural retrieval---resolve entities mentioned in the issue to
symbols, expand along the call graph when gated, and rank by PageRank centrality.
\item \texttt{read\_symbol}: fetch one function/class body by \texttt{file::symbol}, instead of
reading the whole file. This is where \sg{}'s token savings come from.
\item \texttt{expand}: return a symbol's callers and callees, so the agent follows real control
flow rather than guessing files.
\end{itemize}
Each \texttt{search\_code} result is annotated with a one-line, \emph{LLM-free} summary of the
function's purpose---its docstring first line, or a deterministic gloss built from the name,
signature, and body keywords. The agent therefore triages candidates by \emph{what each function
does}, not merely by name or lexical overlap, which the pure rankers (BM25, grep) cannot offer.
In ablation (\texttt{sg-nosummary}) this changes neither file nor function recall---it is not a
\emph{retrieval} signal---but it consistently reduces wasted \texttt{read\_symbol}/\texttt{read\_file}
calls, a contributing (not deciding) reason \sg{} resolves issues at fewer tokens than equally-ranked
baselines.

\paragraph{The method: \sgr{}.} Structural retrieval is precise but low-recall---it finds a
function only when the issue names a resolvable symbol. Lexical BM25 over function bodies is the
opposite: high recall, noisy rank. \sgr{} composes them as \emph{generate-then-rerank}: take
BM25's wide candidate pool (keeping its recall), then reorder it so structurally-confirmed and
issue-named functions rise to the top. This is \emph{not} rank fusion and applies no per-file
cap---the two choices that, in ablation, cost a fusion variant its recall. The result inherits
BM25's recall and \sg{}'s rank, at \sg{}'s token cost.

\section{Evaluation Methodology}
\paragraph{Harness.} A minimal ReAct loop over a fixed substrate: one model, the task issue,
and \texttt{edit\_file}/\texttt{submit} verified by the official SWE-bench harness. Each
pipeline contributes its \emph{native} retrieval/navigation tools; baselines use
\texttt{search\_code}+\texttt{read\_file}. Workspaces are isolated per (task, arm) so no state
or cache leaks between runs or into the submitted patch.
\paragraph{Arms.} Method: \texttt{sg}, \sgr{}. Baselines: \texttt{bm25}, \texttt{grep},
\texttt{hybrid} (BM25$\cup$dense+rerank), \texttt{none}. Competitors (native tools):
\texttt{cbmem}, \texttt{aider}, \texttt{graphify}.
\paragraph{Benchmarks \& models.} We report SWE-bench Verified (100 tasks) with
nemotron-3-super-120B here. A larger Verified split, the contamination-controlled multi-language
SWE-bench Pro, and a Qwen2.5-Coder-32B replication are in progress and reported as ongoing work
(\S\ref{sec:limitations}).
\paragraph{Metrics.} Resolve rate (pass@1, official verifier); input tokens (uncached, uniform
across arms); cost (uncached, and with static-prefix caching); turns; file recall and
precision; \emph{function-level} recall@10 / hit-rate / rank against gold functions recovered by
mapping each patch's changed lines to its enclosing AST function; McNemar's exact test for
paired pass@1.

\section{Analysis: why the token claims do not survive}
\label{sec:analysis}
\paragraph{The loop recovers per-query savings.} Retrieval quality and solve rate are decoupled
(Table~\ref{tab:main}): the best localizers (\sgr{}, BM25) are not the best solvers, and
zero-localization arms (\texttt{none}, aider) solve as well as any. The reason is mechanistic:
when first-search localization misses, the agent searches and reads again until it succeeds, so
\emph{total} tokens depend on how the agent reads, not on a single query's compactness. This is
why aider, whose per-query map is ``compact,'' is in fact the most expensive arm---it pays the
map on every turn. We additionally report cost under static-prefix caching (Figure~\ref{fig:cost}), which roughly
halves aider's cost ($\$0.319\rightarrow\$0.140$) but leaves token \emph{counts} unchanged and
does not alter the ranking: the \sg{} family remains cheapest in both regimes.

\begin{figure}[t]\centering
\includegraphics[width=.86\linewidth]{figures/fig3_cost.pdf}
\caption{Per-task cost, uncached vs.\ static-prefix-cached, sorted by token count. Caching narrows
but does not reverse the ordering; the \sg{} family is cheapest under both. Aider is clipped for
scale ($\$0.32$ uncached).}
\label{fig:cost}
\end{figure}
\paragraph{File is not function.} File-level recall is high for most arms ($\sim$0.85), but
function-level recall is where the headroom is ($\sim$0.4 at best): pipelines routinely find the
right file and the wrong function (Figure~\ref{fig:retrieval}). \sg{}'s function-granular index
and \sgr{}'s rerank are what move this metric---\sgr{} leads on both bars, and is the only arm
whose function recall ($.404$) approaches its file recall.

\begin{figure}[t]\centering
\includegraphics[width=.86\linewidth]{figures/fig2_retrieval.pdf}
\caption{Retrieval quality decomposed into file recall (left bar) and function recall@10 (right
bar) for every arm that performs ranked retrieval. Most arms localise the file but not the
function; the gap is the headroom. \sgr{} leads both. \texttt{none} and aider perform no ranked
retrieval and are omitted.}
\label{fig:retrieval}
\end{figure}
\paragraph{The LLM-graph competitor is not zero-cost.} Graphify requires an LLM call to construct
its graph (its CLI refuses to extract without an API key), a cost a per-task agent re-pays on
every repository; \sg{} builds its index purely with tree-sitter. When this construction cost is
attributed (Table~\ref{tab:build}), the ``token reduction'' framing inverts: Graphify spends
$230$k input $+\,6.6$k output tokens \emph{per repository} to build its graph ($\$0.056$), so its
true per-task cost is $\$0.112$---more than double \sg{}'s $\$0.051$, which has no build term at
all. Codebase-Memory, by contrast, builds its graph with tree-sitter alone (no LLM), so like
\sg{} it carries no build term---yet \sg{} still retrieves more precisely at lower token cost
(Table~\ref{tab:main}), which isolates \emph{retrieval quality}, not build cost, as the
differentiator against a fair zero-LLM structural competitor.

\paragraph{Contamination is the confound.} The strength of \texttt{none} is consistent with
models localizing from memorized, pre-cutoff repositories. Figure~\ref{fig:decoupling} makes the
decoupling explicit: function recall varies by more than $2\times$ across arms ($.40$ down to
$0$) while pass@1 stays pinned at the no-retrieval baseline ($\sim$37\%). We therefore treat
tokens and retrieval quality (deterministic, contamination-free) as the headline, and pass@1 as
``comparable.'' SWE-bench Pro results (in progress) will test whether retrieval value emerges once
localization is no longer memorizable.

\paragraph{Prompt-level symbol leakage.}
Contamination explains why \emph{any} arm can solve $\sim$37\% of tasks; a separate
mechanism inflates the absolute retrieval numbers across the board.
SWE-bench issues are real GitHub reports, and a large fraction name the buggy function
or file path explicitly---in tracebacks (\texttt{File "django/forms/models.py", line 543,
in modelformset\_factory}), code snippets, or issue prose.
Every retrieval arm receives the same issue text, so any lexical system---BM25, grep,
\sg{}, cbmem, graphify---can match these literals verbatim, without structural inference.
This is \emph{not} a bug in the comparison: it is the same information a real developer
(or a real agent) would have, and all arms are identically advantaged.
The implication for \emph{relative} comparisons is that they remain valid: \sg{}'s
recall@1 advantage over BM25 and grep cannot be attributed to leakage because both
systems see the same prompt; where \sg{} wins, it is because structural ranking
orders the named symbol above its lexical neighbours (test files, similarly-named
functions in other modules) rather than merely finding it.
The tasks most discriminative of retrieval quality are those where the issue
describes behaviour without naming the symbol; the most ``easy'' are those with full
tracebacks. \tbd{Add leakage-stratified analysis: split the 100 tasks into
``issue names gold symbol/file'' vs.\ ``behaviour-only'' subsets and report
recall@1 and solve rate separately for each group; confirm SG gap persists on
the harder, name-free subset.}
This applies to all arms in our study (BM25, grep, \sg{}, cbmem, graphify, aider);
none is uniquely disadvantaged or advantaged by prompt leakage relative to the others.

\section{Ablations}
We test seven alternative ways to use the same structural index, all sharing \sg{}'s native
tools and substrate, so any difference is attributable to \emph{how} the index is queried, not to
the index itself (Table~\ref{tab:abl}). Three families emerge.

\textbf{Graph traversal} (\texttt{sg-chain}: rank by call-graph expansion from issue-named
symbols) achieves the highest \emph{file} recall of the family ($.902$) but the lowest pass@1
($42.0\%$) and a function recall ($.312$) below the \sgr{} reranker---it pulls in structurally
adjacent but topically irrelevant neighbours.

\textbf{Semantic / dense} retrieval, using a code-specialised embedder
(\texttt{jina-embeddings-v2-base-code}, $768$-d) five different ways---dense rescoring of the
candidate pool (\texttt{sg-embed}), reciprocal-rank fusion of lexical and dense
(\texttt{sg-hybrid-fusion}), dense rerank of the BM25 pool (\texttt{sg-dense-rerank}),
keyword-conditioned dense (\texttt{sg-keyword-dense}), and search over LLM-free intent summaries
(\texttt{summary-dense})---uniformly \emph{lowers} function recall ($.211$--$.290$) relative to
the purely lexical-plus-structural \sgr{} ($.404$). Embeddings retrieve concept-level neighbours;
they do not localise the specific function a patch must edit. \texttt{sg-dense-rerank} is in fact
the worst arm among these index-use variants on pass@1 ($37.0\%$ in the ablation run).

\textbf{Symbol seeding} (\texttt{sg-seed}: extract symbols named in the issue's traceback frames
and pin them to the top) matches \texttt{none} on pass@1 (both $44.0\%$ in the ablation run) but its hard-pinned seeds
\emph{displace} better structural candidates, leaving recall ($.804$) below the plain core
($.854$). Naive extraction is insufficient; confidence-weighted seeding is left to future work.

No variant beats \sgr{} (Figure~\ref{fig:ablation}). The consistent lesson across all three
families is that structure is best deployed as a \emph{reranker over lexical recall}---not as
standalone graph traversal, not as a semantic retriever, and not as a hard seed.

\section{Deployment: \sg{} in a Production Agent}
\label{sec:deploy}
The controlled harness isolates retrieval by giving every arm the \emph{same} minimal tool set,
so \texttt{search\_code} is the only way to find code. A production agent is different: it keeps
its native grep/glob/read tools and is heavily trained to use them. We ship \sg{} as an MCP
server with editor hooks and run it inside Claude Code to see what survives this shift. Two
observations sharpen the paper's thesis.

\paragraph{(i) For a strong agent, the gain is efficiency, not solve rate.} The agent consults
\sg{} to orient, then frequently navigates natively; it resolves the task with or without the
index, and edits the correct file in essentially every run. \sg{}'s contribution therefore
appears as fewer turns and a smaller \emph{peak context window}---the high-water mark of tokens
the model attends to in a single turn, which the structural index holds to a few tens of
thousands of tokens even on large repositories---rather than as a higher solve rate. This is the
same retrieval/solve-rate decoupling we measure in the harness (\S\ref{sec:analysis}), now
visible in a deployed agent.

\paragraph{(ii) Enforcing retrieval-first is what converts the index into savings.} Because a
capable agent defaults to grep, the per-query token savings of structured retrieval only
materialise if the agent actually queries the index. We add a \emph{retrieval-first gate}: a
\texttt{PreToolUse} hook that denies native text search until the agent has queried \sg{}, then
releases it, with a bounded-fallback so a task the index cannot localise never deadlocks. The
gate is honored even under permission-bypass execution, since hooks are evaluated independently
of the permission prompt. Competing tools (\texttt{cbmem}, \texttt{graphify}, repo-map injection)
expose their index but do not enforce its use; the gate is, to our knowledge, the first such
mechanism. We complement it with two index-quality changes that reduce the agent's fallback to
native tools: demoting test files below implementation in ranked results (so the edit target is
not buried under its tests), and surfacing the edit target's module-level constants inline (so
the agent need not grep for them). Preliminary in-the-loop runs show the gate raises index usage
and lowers native search and turn count; a full Claude-Code evaluation across the task suite is
ongoing and will be reported separately.

% ---------------------------------------------------------------
\section{Limitations and Ongoing Work}
\label{sec:limitations}
Results here are a single model on a single, contaminated benchmark; pass@1 differences are not
significant, and our headline rests on tokens and retrieval quality rather than solve rate. Three
extensions are in progress to test generalization: (i) the contamination-controlled, multi-language
\textbf{SWE-bench Pro} (Go, JavaScript, TypeScript), which our multi-language tree-sitter index
(ten languages: Python, JavaScript, TypeScript, Go, Rust, Java, C\#, C++, Ruby, PHP) is built to
address and which removes the memorization confound; (ii) a \textbf{Qwen2.5-Coder-32B}
replication to confirm the findings are not model-specific; and (iii) the full, at-scale
\textbf{Claude-Code evaluation} of the deployment in \S\ref{sec:deploy} (preliminary in-the-loop
runs reported there). Our function-gold is recovered from patch line spans via AST and is
approximate. We do not claim multi-language generalization beyond the tasks evaluated.

% ---------------------------------------------------------------
\section{Conclusion}
Putting retrieval inside the agent loop changes the conclusion. Token-count reductions, however
large in isolation, do not by themselves make an agent cheaper or better; an imprecise pipeline
is recovered by the loop, and a heavy one is paid for every turn. What helps is \emph{landing
the right function cheaply}. \sg{}'s structural reranking and function-level fetch do this at the
lowest cost of any pipeline we tested, with no LLM in its indexing path---and it is the right way
to evaluate such tools that we hope is the more durable contribution.
