# SkeletonGraph v3 — Evaluation Strategy & Priority Plan

## Publication Priority Order

The goal is **maximum reach** with the first published results. We target platforms and audiences in this order:

### Wave 1: First Publish (Week 1) — GitHub Copilot + `sg prompt` (UI LLMs)
**Why GitHub Copilot first:**
- The most widely used IDE coding agent by a massive margin.
- Copilot agent logs provide session total tokens, MCP tool usage, and LLM turns.
- Proves integration with the largest developer ecosystem (VS Code + Copilot).
- MCP support allows for clean, native-feeling integration.

**Why `sg prompt` (UI LLMs) alongside:**
- Zero integration friction — user pastes context into Claude.ai/ChatGPT/Gemini
- Proves the core thesis without requiring any IDE plugin
- Massive audience: every developer who uses a chat LLM for coding
- Token savings are trivially measurable (count tokens in/out)

**Wave 1 Metrics to publish:**
| Metric | Source | How |
|:---|:---|:---|
| Token reduction ratio (>5x) | `hit_log.jsonl` + Copilot Agent Log | Manually extracted from log |
| Correctness (test pass rate) | `native.patch` vs `sg.patch` against tests | Automated comparison |
| Mode routing accuracy | Expected vs actual mode in hit_log | Auto-compared |
| Modifier firing accuracy | Expected modifier in TASKS vs hit_log | Auto-compared |
| Hit rate (zero additional tool calls) | `tool_calls_after` in hit_log / Copilot log | Cross-referenced |

**Wave 1 Task subset (15 tasks):**
- 5 CODE_FIX/FAST (single function, prove FAST mode works)
- 5 CODE_FIX/STANDARD (multi-function, prove core retrieval)
- 3 PLANNING (prove no-code-body mode saves tokens)
- 2 REVIEW (prove session memory works)

### Wave 2: IDE Agents (Week 2-3) — Cursor + Codex
**Why Cursor second:**
- Largest independent IDE coding agent user base.
- Tier B: session-level token totals from UI.

**Why Codex third:**
- Growing rapidly, CLI-based → semi-automatable
- `AGENTS.md` integration is simple and demonstrable

**Wave 2 Task subset (10 tasks per agent):**
- 3 CODE_FIX/FAST
- 3 CODE_FIX/STANDARD
- 2 PLANNING
- 2 REFACTOR (prove BLAST_FIRST modifier)

### Wave 3: Remaining Agents (Week 3-4) — Claude Code + Antigravity
**Claude Code (Tier A):** Fully automated, per-turn metrics. Scheduled for later when access is procured.
**Antigravity (Tier C):** Qualitative + partial metrics.

## UI LLM Evaluation (Claude.ai / ChatGPT / Gemini)

This is the `sg prompt` flow — no IDE, no MCP, just clipboard.

### How It Works
```
User runs: sg prompt "fix the content-length bug in GET requests"
→ SG assembles optimized context (FAST mode, ~950 tokens)
→ Copies to clipboard
→ User pastes into Claude.ai / ChatGPT / Gemini
→ LLM reads the pre-assembled context and responds
```

### How to Evaluate
1. **Baseline:** Open the repo in the LLM chat, manually paste the problem statement, then manually paste relevant files when the LLM asks for them. Count total tokens sent.
2. **SG run:** Run `sg prompt`, paste the assembled context. Count total tokens sent.
3. **Measure:** Baseline tokens vs SG tokens. Correctness via manual review.

### Commands for UI LLM Evaluation
```powershell
# Setup
cd eval/runs/ui_llm/requests-1142/repo
sg build
sg init

# Baseline: manually explore and paste files into Claude.ai
# Record: how many files did you paste? Total characters/tokens?

# SG run:
sg prompt "fix the content-length bug in GET requests"
# → Clipboard now has the assembled context
# Paste into Claude.ai / ChatGPT
# Record: SG reported token count, response quality
```

### Metrics for UI LLM
| Metric | How to Measure |
|:---|:---|
| Files manually pasted (baseline) | Count during baseline run |
| Total tokens (baseline) | Character count ÷ 4 |
| SG tokens delivered | `sg prompt` output shows token count |
| Reduction ratio | Baseline tokens ÷ SG tokens |
| Response correctness | Manual: did the LLM fix the bug? |
| Modifier adherence | Manual: did the LLM follow BLAST_FIRST/BRAINSTORM instructions? |

## Per-Task Runbook Structure

Each task directory gets a self-contained `RUNBOOK.md` with:
1. **Full problem statement** (copy-paste ready, not truncated)
2. **All commands** for that specific task × agent combination
3. **Expected outcomes** (mode, modifier, token budget)
4. **Evaluation checklist** specific to the task type

These are auto-generated by `setup_workspaces.py`. The runbooks live at:
```
eval/runs/<agent>/<task_id>/RUNBOOK.md
```
